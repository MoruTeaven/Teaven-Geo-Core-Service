## 新增从属关系检查 API

- 新增 `GET /geo/is-subordinate?descendant=xxx&ancestor=xxx&lang=zh` 端点
- 功能：接收两个 GeoNames ID，检查 `descendant` 是否为 `ancestor` 的下属行政单位（即 ancestor 是否出现在 descendant 的祖先链中）
- 参数名使用 `descendant`/`ancestor`（而非 `a`/`b`），语义更清晰
- 修改了以下文件：
  - `src/db/queries.ts`：新增 `queryParentById` 查询
  - `src/services/geo.ts`：新增 `isSubordinate` 业务函数
  - `src/index.ts`：新增路由 `/geo/is-subordinate`
  - `src/types.ts`：新增 `SubordinateResponse` 类型
  - `README.md`：补充 API 文档

## 修复 /geo/resolve 404 问题

- **根因**：`/geo/resolve` 只注册了 `router.post()`，GET 请求命中通配路由返回 404
- **修复 1**：在 `src/index.ts` 中新增 `router.get('/geo/resolve', ...)` 处理函数，从 URL query string 读取 `path` 和 `lang` 参数；同时 POST 也支持非 JSON 的 form/query string 传参
- **修复 2**：在 `src/services/geo.ts` 的 `resolvePath` 中，将模糊匹配的条件从 `if (!result && i > 0)`（跳过第一级）改为 `if (!result)`（第一级也走模糊匹配），解决了单 token 如 "定陶" 因 `parent_id IS NULL` 约束查不到结果的问题。第一级 token 模糊匹配直接取第一个候选；非第一级仍优先匹配 parent_id

## 修复 /geo/search 搜索接口返回空结果

- **根因**：搜索接口查询 `location_search` 表，但 `process-geoname.js` 数据导入脚本从未向该表写入数据。该表在 schema 中仅标注"未来扩展"预留，实际为空。
- **修复**：将 `src/index.ts` 中的搜索 SQL 改为直接从 `location_names` 表模糊匹配 `name` 字段（`ln.name LIKE ?1`），移除对 `location_search` 表的依赖
- **附加优化**：加入 `l.is_active = 1` 过滤 + `ORDER BY ln.name ASC` 排序
